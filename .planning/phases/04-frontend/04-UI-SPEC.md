## Interaction Design

### User Flow (Happy Path)

```
[Page Load]
    → User sees: input field (URL) + submit button
    → UX-04 hint visible: "Vídeos com mais de 15 minutos não são aceitos."

[User pastes URL + clicks submit]
    → Input and button disabled immediately
    → Progress indicator becomes visible
    → State: SUBMITTING → POST /jobs

[POST /jobs → 202 {job_id}]
    → State transitions to POLLING
    → setInterval(2000ms) starts polling GET /jobs/{job_id}

[Polling: status = "queued"]
    → Progress label: "Na fila..."

[Polling: status = "downloading"]
    → Progress label: "Baixando..."
    → sub-stage "checking_duration" → label: "Verificando duração..."
    → sub-stage "downloading" → label: "Baixando áudio..."

[Polling: status = "converting"]
    → Progress label: "Convertendo para WAV..."

[Polling: status = "analyzing"]
    → Progress label: "Analisando BPM e tonalidade..."

[Polling: status = "done"]
    → clearInterval (polling stops)
    → Result card becomes visible
    → Compute WAV size estimate client-side
    → Show: BPM, BPM÷2, BPM×2, key, Camelot, file size, download button
    → Input re-enabled for new submission

[User clicks Download button]
    → Browser follows <a href="/files/{job_id}" download> — no JS redirect
    → WAV file saved as "soundgrabber_{job_id[:8]}.wav"
```

### JS State Machine

States and valid transitions:

```
IDLE
  → SUBMITTING       (user clicks submit with valid input)

SUBMITTING
  → POLLING          (POST /jobs returns 202)
  → ERROR_VALIDATION (POST /jobs returns 422)
  → ERROR_RATE_LIMIT (POST /jobs returns 429)

POLLING
  → POLLING          (GET /jobs/{id} returns queued/downloading/converting/analyzing)
  → DONE             (GET /jobs/{id} returns done)
  → ERROR_JOB        (GET /jobs/{id} returns failed)
  → ERROR_TIMEOUT    (180s elapsed without done/failed)

DONE
  → IDLE             (user clears field or pastes new URL)

ERROR_VALIDATION
  → SUBMITTING       (user edits URL and resubmits)

ERROR_RATE_LIMIT
  → SUBMITTING       (countdown reaches 0; button re-enables automatically)

ERROR_JOB
  → SUBMITTING       (user clicks "Tentar novamente" button)

ERROR_TIMEOUT
  → IDLE             (user dismisses or starts fresh)
```

### State-to-UI Mapping

| State | Input | Submit Button | Progress Indicator | Result Card | Error Area |
|-------|-------|---------------|--------------------|-------------|------------|
| IDLE | enabled | enabled | hidden | hidden | hidden |
| SUBMITTING | disabled | disabled (spinner text) | hidden | hidden | hidden |
| POLLING | disabled | disabled | visible + stage label | hidden | hidden |
| DONE | enabled | enabled | hidden | visible | hidden |
| ERROR_VALIDATION | enabled (highlighted) | enabled | hidden | hidden | visible (inline) |
| ERROR_RATE_LIMIT | disabled | disabled (countdown) | hidden | hidden | visible |
| ERROR_JOB | enabled | hidden | hidden | hidden | visible + retry button |
| ERROR_TIMEOUT | enabled | enabled | hidden | hidden | visible |

### Polling Implementation Contract

- Interval: 2000ms fixed (CONTEXT.md D-01)
- Timeout: 180s = 90 poll cycles (CONTEXT.md D-02)
- Stop on: `status === "done"` OR `status === "failed"` (CONTEXT.md D-03)
- Stop on timeout: clearInterval + show ERROR_TIMEOUT message
- Implementation: `const pollTimer = setInterval(pollStatus, 2000)` with `clearInterval(pollTimer)` in done/failed/timeout branches

### Rate-Limit Countdown Contract

- Source: `Retry-After` response header (integer, seconds)
- Countdown: `setInterval(1000, decrement)` — local decrement, not server-synced (CONTEXT.md specific ideas)
- Button state: disabled with text showing remaining seconds ("Tente novamente em 47s")
- On zero: clearInterval, re-enable submit, clear countdown display (CONTEXT.md D-05)

### WAV Size Estimation Contract

- Trigger: when polling returns `status === "done"` and `duration_sec` is present
- Formula: `Math.round(duration_sec * 44100 * 2 * 2 / 1_000_000)` (CONTEXT.md D-08)
- Display: prefix `~` always (estimate, not exact)
- Format: if result >= 10 → `~{N} MB` (0 decimal); if result < 10 → `~{N.N} MB` (1 decimal)
- Position: inside result card, before download button (CONTEXT.md D-09)

### Download Mechanism Contract

- Element: `<a>` tag with `href="/files/{job_id}"` and `download` attribute
- Filename served by server: `soundgrabber_{job_id[:8]}.wav`
- No JS redirect, no fetch — native browser anchor download
- Source: `api/main.py:download_file()` returns `FileResponse` with filename set

## API Consumption Contract

For completeness — what the JS must handle from each endpoint.

### POST /jobs

Request: `{ "youtube_url": "..." }` — **campo obrigatório: `youtube_url`, NÃO `url`** (ver RESEARCH.md Pitfall 1)
Success (202): `{ "job_id": "{uuid}" }` → start polling
Error (422): `{ "error": str, "error_type": str }` → ERROR_VALIDATION state
Error (429): header `Retry-After: {N}` → ERROR_RATE_LIMIT state + countdown

### GET /jobs/{job_id}

Poll every 2000ms. Handle each shape:

| `status` | Additional fields | JS action |
|----------|------------------|-----------|
| `"queued"` | — | update progress label, continue polling |
| `"downloading"` | `stage: str\|null` | update progress label, continue polling |
| `"converting"` | `stage: str\|null` | update progress label, continue polling |
| `"analyzing"` | `stage: str\|null` | update progress label, continue polling |
| `"done"` | `bpm, bpm_half, bpm_double, key, camelot, duration_sec, download_url` | stopPolling, showDone |
| `"failed"` | `error: str, error_type: str` | stopPolling, showErrorJob |

### GET /files/{job_id}

Not called by JS directly. Used as `href` in `<a download>` anchor.
`download_link.href = '/files/' + jobId` set when status is `"done"`.
