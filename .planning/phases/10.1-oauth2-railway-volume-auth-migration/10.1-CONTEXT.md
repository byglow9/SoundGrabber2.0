# Phase 10.1: OAuth2 + Railway Volume Auth Migration — Context

**Gathered:** 2026-05-12
**Status:** Ready for planning

<domain>
## Phase Boundary

Substituir a camada de autenticação do yt-dlp: de cookies de browser + serviço bgutil PO Token
para OAuth2 device flow com token persistido em Railway Volume.

Entregáveis concretos:
1. `pipeline.py` e `api/` usam OAuth2 (`username=oauth2`, `cachedir`) — sem cookies, sem bgutil
2. Railway Volume montado em `/data/yt-dlp-cache` — token sobrevive a restarts e redeploys
3. Serviço bgutil deletado do Railway + `bgutil-ytdlp-pot-provider` removido de requirements.txt
4. Operador faz device auth **uma vez** via `railway run`; a partir daí, token auto-renova

Fora do escopo:
- Mudanças no fluxo de jobs (API, Celery, análise BPM/key) — pipeline lógico não muda
- Mudanças no frontend
- Novos endpoints

</domain>

<decisions>
## Implementation Decisions

### D-01 — Estratégia de migração: quebra limpa

Remover **toda** a lógica de cookies e bgutil do código. Sem suporte duplo, sem fallback para
cookies se OAuth2 falhar. Se `YTDLP_CACHE_DIR` não estiver configurado, o app loga CRITICAL
e continua (mesmo padrão da probe de cookies atual em PIPE-05).

Rationale: suporte duplo duplica complexidade de config e testes sem benefício real. OAuth2
é superior em todas as dimensões; manter cookies gera código morto.

### D-02 — Assinaturas de função após a migração

```python
# pipeline.py — antes:
check_duration(url, cookies_path, bgutil_base_url) -> dict
download_audio(url, cookies_path, po_token, bgutil_base_url) -> Path

# pipeline.py — depois:
check_duration(url, cache_dir) -> dict
download_audio(url, cache_dir) -> Path
```

`cache_dir` é o path do Railway Volume (`/data/yt-dlp-cache`), lido de `YTDLP_CACHE_DIR`.

### D-03 — Opções yt-dlp com OAuth2

Substituir nos `ydl_opts` de `check_duration` e `download_audio`:

```python
# Remover:
"cookiefile": cookies_path
"extractor_args": {"youtube": ["player_client=web", "getpot_bgutil_baseurl=..."]}
"po_token": ...

# Adicionar:
"username": "oauth2"
"password": ""
"cachedir": cache_dir
```

OAuth2 usa cliente diferente do `web` — PO Token não é requerido. Sem `extractor_args`
especial necessário (yt-dlp native OAuth2 gerencia o client type internamente).

### D-04 — Remoção da probe PIPE-06

`BgutilUnavailable` e o probe `httpx.get(bgutil_base_url)` em `download_audio` são
removidos inteiramente. Sem bgutil, sem probe. `api/tasks.py` remove o `except BgutilUnavailable`.

### D-05 — Teardown do bgutil

- **Código:** remover `bgutil-ytdlp-pot-provider==0.8.5` de `requirements.txt`; remover
  `BgutilUnavailable`, `bgutil_base_url` de `pipeline.py`; remover `bgutil_base_url` de
  `api/config.py` e `api/tasks.py`
- **Railway:** deletar serviço `bgutil` (ID: `2fc3a8a5-b50b-48e2-b408-87167f5ac28a`) via MCP
- **Env vars removidas:** `BGUTIL_BASE_URL`, `YTDLP_COOKIES_B64`
- **Env var adicionada:** `YTDLP_CACHE_DIR=/data/yt-dlp-cache`

### D-06 — Railway Volume

- Montar Railway Volume no serviço "Uvicorn Web + Celery Worker" em `/data/yt-dlp-cache`
- Setar `YTDLP_CACHE_DIR=/data/yt-dlp-cache` nas env vars do serviço
- O Celery Worker (mesmo container via `start-all.sh`) herda o Volume automaticamente
- Permissão do diretório: `chmod 700 /data/yt-dlp-cache` no startup

### D-07 — Fluxo de auth inicial (one-time, via `railway run`)

```bash
railway run yt-dlp \
  --username oauth2 \
  --password '' \
  --cache-dir /data/yt-dlp-cache \
  --skip-download \
  "https://www.youtube.com/watch?v=dQw4w9WgXcQ"
```

yt-dlp imprime um código + URL (`google.com/device`). Operador entra no link, aprova.
Token gravado em `/data/yt-dlp-cache/youtube-oauth2.token_data` (Railway Volume).
Todos os containers subsequentes carregam o token desse Volume.

O plano deve incluir um **checkpoint humano** para esta etapa com as instruções exatas.

### D-08 — Startup validation (substituto de PIPE-05)

`api/main.py` lifespan: verificar se `settings.cache_dir` está configurado E se
`Path(settings.cache_dir)` existe. Se não existir ou estiver vazio, logar CRITICAL
(não-bloqueante, mesmo padrão do PIPE-05 atual). Mensagem deve mencionar
`railway run yt-dlp --username oauth2 ...` como próximo passo.

### D-09 — Estratégia de testes

- `conftest.py`: `YTDLP_CACHE_DIR` aponta para `tmp_path` (pytest fixture)
- yt-dlp mockado a nível de `yt_dlp.YoutubeDL` (patch) — sem chamadas reais ao Google
- Testes verificam que `ydl_opts` contém `"username": "oauth2"` e `"cachedir"` correto,
  e que `"cookiefile"` e `"extractor_args"` de bgutil **não** estão presentes
- `DEV_MODE=true` já está em `conftest.py` — bypassa Redis auth (não muda nada aqui)
- Nenhuma conta Google necessária em CI

### Claude's Discretion

- Ordem de waves no plano: código primeiro (pipeline.py → config.py → tasks.py → tests),
  depois Railway infra (Volume → env vars → delete bgutil), depois checkpoint de auth.
- Se yt-dlp OAuth2 nativo tiver limitações descobertas durante implementação, usar
  `--username oauth2` via subprocess wrapper é opção válida (não preferida — registrar
  no SUMMARY se usado).

</decisions>
