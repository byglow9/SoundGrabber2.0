### Discoveries da Task 1 (4 bugs novos encontrados durante execução)

**Bug 1 — `YTDLP_NO_PLUGINS=1` silencioso (mais crítico)**
`start-all.sh` exportava `YTDLP_NO_PLUGINS=1`, adicionado em fase anterior para "limpar plugins antigos". Isso desativava **todos** os plugins do yt-dlp, incluindo o `bgutil-ytdlp-pot-provider`. Sem esse export, GetPOT carrega normalmente. Removido em `c1e5657`.

**Bug 2 — Formato errado de `extractor_args` na Python API do yt-dlp 2026**
O código inicial usava formato CLI (lista de strings): `["player_client=web", "getpot_bgutil_baseurl=url"]`. A Python API do yt-dlp 2026 exige dicionário aninhado: `{"player_client": ["web"], "getpot_bgutil_baseurl": ["url"]}`. Com formato errado, o yt-dlp ignorava os parâmetros e usava clientes padrão (`android_vr`), para os quais bgutil 0.8.x não tem provider. Corrigido em `4d88510`.

**Bug 3 — Version mismatch plugin vs servidor**
`requirements.txt` tinha `bgutil-ytdlp-pot-provider==0.8.5`, mas o servidor bgutil no Railway está em `0.8.1`. Isso causava HTTP 400 na API do servidor ao pedir PO Token. Alinhado para `==0.8.1` em `3635498`.

**Bug 4 — `player_client=web` não suportado isolado pelo bgutil 0.8.x**
bgutil 0.8.x suporta `web_safari` e `web`. Usar apenas `["web"]` causava tentativa com `android_vr` nos fallbacks do yt-dlp, para os quais `No player PO Token provider available`. Corrigido para `["web_safari", "web"]` em `7fabeb8`.

### Diagnóstico do bloqueio atual

O stack técnico está correto e funcionando:
- ✅ Cookies chegam ao container e são escritos no Volume
- ✅ yt-dlp reconhece cookies como autenticados (`Found YouTube account cookies`)
- ✅ Node runtime disponível (`node-24.15.0`)
- ✅ Plugin bgutil carregado (YTDLP_NO_PLUGINS removido)
- ✅ GetPOT gera PO Token com sucesso (HTTP 200 do servidor bgutil)
- ✅ PO Token enviado ao yt-dlp via extractor_args correto
- ❌ YouTube retorna `LOGIN_REQUIRED` mesmo com PO Token + cookies

**Causa mais provável: cookies do Volume expirados/inválidos.**

O arquivo `www.youtube.com_cookies.txt` que foi base64-encodado para `YTDLP_COOKIES_B64` e escrito no Volume pode ter tokens já rotacionados pelo Google. O `__Secure-3PSID` e companhia duram semanas a meses; se expirados, o YouTube vê a sessão como inválida independente do PO Token. O yt-dlp disse `Found YouTube account cookies` mas não necessariamente validou os tokens — apenas encontrou o arquivo.

**Evidência adicional:** os bytes do cookies.txt no Volume caíram de 2987 para 1558/1637 durante tentativas — indica que o yt-dlp sobrescreveu o jar com cookies reduzidos (comportamento de invalidação de sessão).

**1. bgutil-ytdlp-pot-provider pinado em 0.8.1, não 0.8.5**
O plano especificava `==0.8.5`. O servidor Railway está em `0.8.1` (0.8.5 não existe no PyPI). Ajustado para alinhar com o servidor.

bgutil 0.8.x suporta ambos. `web_safari` é o cliente autenticado padrão do yt-dlp — mais provável de ter path de autenticação correto.

**3. start-all.sh modificado (não estava no escopo original)**
`YTDLP_NO_PLUGINS=1` foi descoberto durante diagnóstico como bloqueador crítico. Remoção necessária.

**4. Formato de extractor_args invertido (Python API vs CLI)**
O plano especificava formato CLI de strings. A Python API 2026 do yt-dlp requer nested dict. Corrigido — sem impacto no comportamento, mas necessário para os parâmetros serem lidos.
