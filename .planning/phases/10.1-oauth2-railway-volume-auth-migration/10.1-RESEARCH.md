## Summary

Esta fase tem dois objetivos distintos: (1) eliminar a dependência do bgutil PO Token service e (2) criar persistência de autenticação no Railway Volume. O objetivo 2 pode ser alcançado, mas **não via OAuth2** — que foi removido do yt-dlp. A abordagem viável é persistir os cookies do YouTube no Railway Volume em vez de codificá-los como env var Base64, eliminando a necessidade de re-deploy quando os cookies expiram.

**O que muda na arquitetura:** Em vez de `username=oauth2`/`cachedir`, a solução usa `cookiefile` apontando para `/data/yt-dlp-cache/cookies.txt` — um arquivo no Railway Volume que persiste entre restarts. O operador faz o processo de atualização de cookies uma vez e grava no Volume; a partir daí, o app carrega do Volume automaticamente.

O teardown do bgutil permanece intacto: remover `bgutil-ytdlp-pot-provider` de requirements.txt, deletar o serviço Railway, remover as env vars. O yt-dlp 2026.3.17 usa `('ios', 'mweb')` como clientes padrão sem autenticação — sem necessidade de PO Token para `ios` e `mweb` [VERIFIED: INNERTUBE_CLIENTS no código instalado]. Com cookies válidos, o cliente autenticado usa `('web_creator', 'mweb')` [VERIFIED: `_DEFAULT_AUTHED_CLIENTS`].

**Primary recommendation:** Adaptar D-03 para persistir `cookiefile=/data/yt-dlp-cache/cookies.txt` (Railway Volume) em vez de `username=oauth2`/`cachedir`. O plano deve registrar explicitamente esta adaptação como substituição do CONTEXT.md D-03 e obter validação do usuário antes de implementar.

def check_duration(url: str, cache_dir: str) -> dict:
    ydl_opts: dict[str, Any] = {
        "quiet": True,
        "no_warnings": True,
        "skip_download": True,
        "socket_timeout": 30,

### Padrão de startup validation (D-08 — adaptado)

```python
# api/main.py — _check_oauth_cache() adaptado para cookies no Volume
def _check_oauth_cache(cache_dir: str) -> None:
    """PIPE-05 replacement: verifica cookies.txt no Railway Volume."""
    if not cache_dir:
        logger.critical(
            "AUTH: YTDLP_CACHE_DIR não configurado. Downloads podem falhar com bot detection. "
            "Monte um Railway Volume em /data/yt-dlp-cache e configure YTDLP_CACHE_DIR."
        )
        return
    cookies_file = Path(cache_dir) / "cookies.txt"
    if not cookies_file.exists():
        logger.critical(
            "AUTH: cookies.txt não encontrado em %s. "
            "Copie cookies válidos do YouTube para %s no Railway Volume. "
            "Exports podem ser feitos via extensão 'Get cookies.txt LOCALLY' no browser.",
            cache_dir, cookies_file,
        )
        return
    # Verificação do sentinel __Secure-3PSID (mesmo padrão do PIPE-05)
    try:
        content = cookies_file.read_text(encoding="utf-8", errors="ignore")
    except OSError as e:
        logger.critical("AUTH: Não foi possível ler %s: %s", cookies_file, e)
        return
    if "__Secure-3PSID" not in content:
        logger.critical(
            "AUTH: cookies.txt em %s não contém '__Secure-3PSID'. "
            "Cookie pode estar expirado. Re-exporte do browser.",
            cookies_file,
        )
```

### Anti-Patterns a Evitar

- **Hardcode de `/data/yt-dlp-cache/cookies.txt`:** Usar sempre `settings.cache_dir` + `cookies.txt` — Railway Volume mount pode mudar
- **Remover httpx de requirements.txt sem verificar:** httpx é usado no probe bgutil que será removido, mas pode ser dependência indireta de outro módulo
- **chmod 600 no startup.sh antes do Volume estar montado:** Volumes só são montados em runtime, não durante build (Railway docs). O chmod deve ser no lifespan ou no start-all.sh após startup

## Don't Hand-Roll

| Problema | Não Construir | Usar Em Vez | Por Que |
|---------|---------------|-------------|---------|
| Persistent cookies no servidor | Storage customizado, serialização própria | Netscape cookies.txt + Railway Volume | Formato padrão que yt-dlp lê nativamente; Volume persiste entre restarts/redeploys |
| Expiração de token | Sistema de refresh customizado | Re-export manual de cookies pelo operador | Cookies YouTube duram semanas/meses; refresh automático requer conta Google no servidor (risco de ban) |
| Detecção de cookies expirados | Parser do Netscape format | Verificar `__Secure-3PSID` sentinel (padrão PIPE-05 já existente) | Cookie expirado ainda tem o token mas inválido; o sentinel é o indicador mais confiável |

## Common Pitfalls

### Pitfall 1: OAuth2 inexistente no yt-dlp 2026
**O que dá errado:** Implementar D-03 como escrito (username=oauth2, cachedir) resulta em `DownloadError: Login with OAuth is no longer supported` em runtime — não em tempo de import, não em tempo de config, mas quando `extract_info()` é chamado.
**Por que acontece:** yt-dlp 2024.11.18 removeu o suporte OAuth2 por mudanças no Google. O código instalado (2026.3.17) tem o hard reject explícito.
**Como evitar:** Substituir D-03 por abordagem de cookies no Volume (descrita acima) — mesmo resultado de persistência, arquitetura diferente.
**Sinais de alerta:** Se `test_pipe_oauth2_in_ydl_opts` passar mas o E2E falhar com "Login with OAuth is no longer supported" — o teste mockava YoutubeDL mas não testou o comportamento real.

### Pitfall 2: Volume não montado durante build do Railway
**O que dá errado:** `chmod 700 /data/yt-dlp-cache` em Dockerfile/nixpacks falha ou não persiste porque o Volume é montado em runtime.
**Por que acontece:** Railway monta Volumes no startup do container, não durante o build. Dados escritos em build não persistem no Volume.
**Como evitar:** Fazer o chmod no lifespan do FastAPI ou no `start-all.sh`, não em scripts de build.

### Pitfall 3: Testes verificam `username=oauth2` em ydl_opts mas OAuth2 não funciona em runtime
**O que dá errado:** D-09 especifica testes que verificam `"username": "oauth2"` nos ydl_opts. Com a adaptação de D-03, os testes passam mas o comportamento real está testando a abordagem errada.
**Por que acontece:** YoutubeDL mockado não executa o código de validação do extractor YouTube.
**Como evitar:** Com a adaptação, os testes devem verificar `"cookiefile"` aponta para `Path(cache_dir) / "cookies.txt"` (em vez de `username=oauth2`). A semântica do teste D-09 continua válida: verificar que ydl_opts está correto — só o conteúdo correto muda.

### Pitfall 4: httpx removido de requirements.txt quando ainda é dependência
**O que dá errado:** `bgutil-ytdlp-pot-provider` foi o único consumidor explícito de httpx no código de produção (probe PIPE-06). Remover httpx junto com bgutil pode quebrar se outro módulo faz `import httpx`.
**Por que acontece:** httpx aparece no requirements.txt como dependência explícita do projeto.
**Como evitar:** Verificar se httpx tem outros usos no projeto antes de remover. O probe bgutil usa `pipeline.httpx.get()` — remover esse import específico é suficiente. httpx pode ser mantido em requirements.txt se usado por outros módulos ou como dependência de fastapi/celery.

### Pitfall 5: Deletar serviço bgutil Railway antes do código estar funcional
**O que dá errado:** Se o bgutil for deletado antes do redeploy com o novo código, e o novo código tiver bug, há janela sem auth funcional.
**Por que acontece:** Tentação de fazer tudo de uma vez (código + infra Railway).
**Como evitar:** Ordem de waves: código primeiro → deploy + verificar E2E → deletar bgutil.

### Pitfall 6: test_bgutil_08x_extractor_key_* falham após remoção
**O que dá errado:** `test_pipeline_fixes.py` tem testes que verificam que `getpot_bgutil_baseurl` está presente em `check_duration` e `download_audio`. Após remover o bgutil, esses asserts falham.
**Por que acontece:** Os testes foram escritos para verificar que o bgutil estava corretamente configurado (Phase 8/9/10). Agora precisam verificar que o bgutil foi **removido**.
**Como evitar:** Inverter os asserts: após a migração, os testes devem confirmar que `getpot_bgutil_baseurl` e `cookiefile` (bgutil style) **não** estão no código, e que `cookiefile` com path do Volume está presente.

## Security Domain

Esta fase não adiciona novos endpoints HTTP. Os controles do CLAUDE.md Security Gate aplicam-se às modificações:

### Controles aplicáveis às modificações desta fase

| Controle | Aplicável | Ação |
|---------|----------|------|
| Rate limiting em novos endpoints | Não — nenhum endpoint novo | — |
| Permissões 0o600 em /tmp | Sim — cookies.txt no Volume deve ser 0o600 | `chmod 600 /data/yt-dlp-cache/cookies.txt` no checkpoint |
| Prefixo `sg_` em /tmp | Não se aplica — cookies no Volume `/data/`, não em `/tmp` | — |
| Path traversal em cookiefile | Sim — `cookiefile` path deve ser validado | Construir path como `Path(cache_dir) / "cookies.txt"` sem input do usuário |

### Threat: cookies.txt no Volume lidos por outros processos

O Railway Volume é montado como root. Para proteger cookies.txt (contém sessão Google):
```bash
chmod 700 /data/yt-dlp-cache    # diretório — D-06
chmod 600 /data/yt-dlp-cache/cookies.txt  # arquivo — durante checkpoint humano
```

Não há risco de path traversal porque o caminho de `cookiefile` é construído programaticamente via `Path(settings.cache_dir) / "cookies.txt"` — não vem de input externo.

## Open Questions

1. **Adaptação de D-03: usuário precisa confirmar antes de implementar**
   - O que sabemos: OAuth2 é inválido no yt-dlp 2026.3.17; cookies no Volume é a alternativa técnica mais próxima
   - O que é incerto: se o usuário quer prosseguir com cookies no Volume (mudança de arquitetura em relação ao CONTEXT.md) ou pausar e discutir uma estratégia diferente
   - Recomendação: O planner deve colocar esta questão como **bloqueador da Wave 1** — o plano pode ser escrito mas a Wave 1 (código) só deve ser executada após confirmação do usuário sobre a adaptação de D-03

2. **httpx em requirements.txt**
   - O que sabemos: httpx é usado diretamente em `pipeline.py` para o probe bgutil (será removido)
   - O que é incerto: se fastapi/httpx/celery têm httpx como dependência transitiva que já estava satisfeita, ou se httpx precisa estar explícito
   - Recomendação: manter httpx em requirements.txt mesmo após remover o probe — custo zero, risco zero de quebrar instalação

3. **`railway run` para copiar cookies.txt**
   - O que sabemos: `railway run` executa comandos no container com o Volume montado
   - O que é incerto: se `railway run -- bash -c "cp ..."` aceita arquivo local como argumento ou se o arquivo precisa ser encodado e echado
   - Recomendação: o plano deve incluir dois métodos alternativos para o checkpoint humano (cp via railway run stdin pipe, ou YTDLP_COOKIES_B64 como one-time bootstrap)
