    Criar scripts/pi-setup.sh conforme estrutura abaixo. O script roda como root (sudo bash pi-setup.sh) no Pi do Moisés. Cada seção tem um header comentado em português explicando o WHY.

    Estrutura obrigatória (nesta ordem):

    1. Shebang + set -e + chmod 750 (Security Gate):
       - Linha 1: #!/usr/bin/env bash
       - Linha 2: set -e (aborta em qualquer erro — crítico para não deixar sistema parcialmente configurado)
       - Linha 3: chmod 750 "$(realpath "$0")" (Security Gate: sem world-execute)

    2. Verificação de root: se não for root (id -u != 0), imprimir mensagem clara e sair. Script precisa de sudo para instalar pacotes e editar /boot.

    3. SEÇÃO 1 — Arquitetura (D-01/D-02):
       - Executar ARCH=$(uname -m)
       - Imprimir "[pi-setup] Arquitetura detectada: $ARCH"
       - Se ARCH for "aarch64": imprimir mensagem positiva sobre 64-bit
       - Se ARCH for "armv7l": imprimir aviso (não erro) que o Pi está em 32-bit, mas a fase continua normalmente — Phase 13 cuida da estratégia Docker para arm/v7
       - Nunca fazer exit aqui (D-01: não bloqueante)

    4. SEÇÃO 2 — Docker (por D-09, planner escolhe get.docker.com como mais reproduzível para armhf e aarch64):
       - Idempotente: verificar se `command -v docker` já existe antes de instalar
       - Se não existir: instalar via `curl -fsSL https://get.docker.com | sh`
       - Por que get.docker.com: suporta tanto armhf quanto aarch64 automaticamente, sem precisar selecionar repositório manualmente
       - Após instalar: `usermod -aG docker pi 2>/dev/null || true` (adiciona usuário pi ao grupo docker; ignora erro se usuário não existir)
       - Imprimir a versão do docker após instalação/verificação

    5. SEÇÃO 3 — Swap 2GB (D-05/D-10 — idempotente):
       - Verificar se /swapfile já existe: `[ -f /swapfile ]`
       - Se já existe: imprimir "[pi-setup] Swap já configurado — pulando criação" e continuar
       - Se não existe: fallocate -l 2G /swapfile, chmod 600 /swapfile, mkswap /swapfile, swapon /swapfile
       - Verificar se entrada já existe em /etc/fstab antes de adicionar: `grep -q "/swapfile" /etc/fstab`
       - Se não existe no fstab: adicionar linha `/swapfile none swap sw 0 0`
       - Por que fallocate em vez de dd: muito mais rápido para criar arquivos grandes no Pi

    6. SEÇÃO 4 — Memory cgroups (D-06 — idempotente):
       - Alvo: /boot/firmware/cmdline.txt (Raspberry Pi OS Bookworm; fallback para /boot/cmdline.txt se o primeiro não existir)
       - IMPORTANTE: fazer backup do arquivo antes de modificar: cp /boot/firmware/cmdline.txt /boot/firmware/cmdline.txt.bak
       - Verificar se `cgroup_enable=memory` já está presente: grep -q "cgroup_enable=memory" no arquivo
       - Se já presente: imprimir "[pi-setup] Memory cgroups já configurados — pulando"
       - Se não presente: adicionar `cgroup_enable=memory cgroup_memory=1` ao FINAL da linha existente (não criar nova linha) usando sed
         Usar: sed -i 's/$/ cgroup_enable=memory cgroup_memory=1/' /boot/firmware/cmdline.txt
       - Por que cgroups: sem isso, `mem_limit` no docker-compose é silenciosamente ignorado (P-ARM-08)
       - Imprimir aviso: "[pi-setup] ATENÇÃO: Memory cgroups exigem reboot para ativar — o script continua mas um reboot é necessário ao final"

    7. SEÇÃO 5 — Hardware watchdog (D-03/D-04 — idempotente):
       - Alvo 1: /boot/firmware/config.txt — adicionar `dtparam=watchdog=on`
         Verificar: grep -q "dtparam=watchdog=on" /boot/firmware/config.txt
         Se não presente: echo "dtparam=watchdog=on" >> /boot/firmware/config.txt
       - Alvo 2: /etc/modules-load.d/bcm2835-wdt.conf — arquivo com conteúdo `bcm2835-wdt`
         Idempotente: criar ou sobrescrever (echo "bcm2835-wdt" > /etc/modules-load.d/bcm2835-wdt.conf)
       - Por que hardware watchdog: reinicia o Pi mesmo se o kernel ou systemd travar — única defesa em operação headless remota (P-ARM-07)
       - Por que timeout padrão 15s (D-04): adequado — systemd responde em <2s normalmente

    8. SEÇÃO 6 — log2ram (D-07):
       - Por que log2ram: Pi rodando 24/7 pode queimar o SD em meses de writes contínuos de /var/log (P-ARM-06)
       - Idempotente: verificar `command -v log2ram` antes de instalar
       - Se já instalado: imprimir "[pi-setup] log2ram já instalado — pulando"
       - Se não instalado: instalar via curl do repositório oficial azlux (D-07):
         curl -L https://github.com/azlux/log2ram/archive/master.tar.gz -o /tmp/log2ram.tar.gz
         tar -xzf /tmp/log2ram.tar.gz -C /tmp
         cd /tmp/log2ram-master && bash install.sh
         cd / && rm -rf /tmp/log2ram.tar.gz /tmp/log2ram-master
       - NUNCA usar `curl ... | sh` ou `curl ... | bash` (pipe para shell é eval implícito de input de rede — proibido pelo Security Gate D-09)
       - Alternativa segura: baixar para arquivo local, inspecionar path, executar separado

    9. SEÇÃO 7 — Verificação final (D-11):
       - Imprimir seção de status com separador visual claro
       - Status 1: Arquitetura — imprimir uname -m
       - Status 2: Docker — imprimir docker --version (ou "não instalado" se command -v falhar)
       - Status 3: Swap — imprimir saída de `swapon --show` (ou "nenhum swap ativo" se vazio)
       - Status 4: Cgroups — imprimir linha atual de /boot/firmware/cmdline.txt para Moisés confirmar visualmente que contém cgroup_enable=memory
       - Status 5: Watchdog — imprimir conteúdo de /boot/firmware/config.txt grep por watchdog
       - Status 6: log2ram — imprimir `systemctl is-active log2ram 2>/dev/null || echo "não disponível (pode precisar de reboot)"`
       - Imprimir mensagem final: "[pi-setup] Setup concluído. Reboot necessário para ativar cgroups e watchdog. Execute: sudo reboot"

    Proibições absolutas no script:
       - Sem `eval` em nenhuma forma
       - Sem `curl ... | sh` ou `curl ... | bash` (Security Gate)
       - Sem linha em branco entre `#!/usr/bin/env bash` e `set -e`
       - Sem modificar nada relacionado ao Tailscale
       - Sem mencionar Railway, bgutil, ou partes da aplicação

<threat_model>
## Trust Boundaries

| Boundary | Descrição |
|----------|-----------|
| operador → script | Script executado como root no Pi — input externo mínimo (sem parâmetros aceitos) |
| script → internet | Dois downloads de rede: get.docker.com e GitHub azlux/log2ram tarball |
| script → /boot | Modificações em cmdline.txt e config.txt — erros aqui podem impedir o boot |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-01 | Tampering | curl https://get.docker.com (pipe to sh) | mitigate | NÃO usar pipe direto — baixar e executar separado; get.docker.com é HTTPS verificado e padrão da indústria; risco residual baixo |
| T-12-02 | Tampering | curl https://github.com/azlux/log2ram tarball | mitigate | Baixar para /tmp/log2ram.tar.gz local, depois executar install.sh separado — sem pipe para shell (Security Gate D-09) |
| T-12-03 | Tampering | /boot/firmware/cmdline.txt modificado com sed | mitigate | Fazer backup (cp .bak) antes de modificar; usar sed idempotente (grep antes); qualquer erro em set -e aborta o script |
| T-12-04 | Denial of Service | fallocate -l 2G /swapfile em SD card sem espaço | accept | Script abortará com erro de fallocate se não houver espaço — set -e garante que o sistema não continua em estado inconsistente |
| T-12-05 | Elevation of Privilege | Script requer root (sudo) | accept | Script verifica id -u == 0 explicitamente e sai se não for root; sudo é necessário para instalar Docker e modificar /boot |
| T-12-06 | Repudiation | Sem log persistente da execução | accept | O Moisés documenta manualmente o resultado em 12-SETUP-LOG.md (Plan 02) — log de auditoria explícito |
| T-12-07 | Information Disclosure | Conteúdo de /boot exposto via output do script | accept | Apenas dtparam e cgroup params são impressos; sem credenciais ou tokens em /boot/firmware/config.txt |
</threat_model>
