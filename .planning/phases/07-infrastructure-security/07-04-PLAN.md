<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Internet -> Railway edge | Railway termina TLS, faz HTTP->HTTPS 301, encaminha trafego HTTPS para o container interno |
| Railway internal network -> Redis service | Redis acessivel apenas pela rede privada Railway com senha obrigatoria injetada |
| Operador -> Railway dashboard | Acesso admin ao dashboard requer autenticacao Railway (fora do escopo desta fase) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-7-DEPLOY-01 | Information Disclosure | Logs Railway expondo REDIS_URL completo | accept | Logs Railway sao acessiveis apenas pelo proprietario do projeto; senha eh rotativa pelo Railway. 07-DEPLOY-LOG.md instrui mascarar senhas (`***`). |
| T-7-DEPLOY-02 | Spoofing | Subdomain Railway sequestrado (takeover) | accept | Risco baixo: `*.up.railway.app` eh controlado por Railway; subdomain takeover requer dropping o servico (operador controla isso). |
| T-7-DEPLOY-03 | Tampering | Variavel REDIS_URL trocada manualmente para URL sem senha | mitigate | _check_redis_auth no lifespan rejeita imediatamente; servico fica em "Failed" e nao aceita trafego. Plan 02 testou esse caminho. |
| T-7-DEPLOY-04 | Denial of Service | Celery worker sem REDIS_URL fica idle | mitigate | Pitfall 3 do RESEARCH.md: passos 4 do checkpoint exige REDIS_URL no celery-worker; logs do worker validam conexao no Passo 5. |
| T-7-DEPLOY-05 | Elevation of Privilege | DEV_MODE acidentalmente definido em producao | mitigate | Passo 3 explicita "NAO definir DEV_MODE aqui (D-14)"; auditavel via dashboard Variables. |
</threat_model>
