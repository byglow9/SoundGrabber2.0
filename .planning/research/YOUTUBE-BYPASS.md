# YouTube Download Bypass — Approach Landscape

**Domain:** yt-dlp from datacenter IPs (Railway PaaS)
**Researched:** 2026-05-10
**Milestone:** v1.2 YouTube Pipeline Fix
**Overall confidence:** HIGH for bgutil path, LOW for alternatives

---

## Summary

YouTube's bot detection operates at two independent layers. The **network layer** inspects origin IP reputation: all major cloud provider ranges (AWS, GCP, Railway, DigitalOcean, etc.) are flagged before any cookie or token is evaluated. The **client layer** inspects player client type, PO token presence, and cookie validity.

Solving only one layer is insufficient. The current pipeline uses `android` client with cookies when bgutil is absent, and `web` client + PO token when bgutil is present. The Android-only path fails because Android client + cookies from a datacenter IP now returns thumbnail/image data instead of audio streams. The web path requires a live bgutil sidecar that was removed from Railway.

As of early 2026, the viable option space has narrowed sharply:
- `tv_embedded` was removed from yt-dlp entirely in January 2026
- `tv` client now returns `LOGIN_REQUIRED` from datacenter IPs since January 16, 2026
- `web` client without a valid PO token returns SABR-only formats (not traditional DASH/audio-only streams)
- `ios` client silently ignores cookies, making it useless for authenticated format access
- OAuth2 login is dead (killed by Google in 2024)

**The only reliable path on datacenter IPs is the `web` client + valid PO token auto-generated by a bgutil sidecar.** This is what worked during the one confirmed successful download in the project history.

## Approach 1: bgutil HTTP Sidecar (web client + PO Token)

### Mechanism

bgutil-ytdlp-pot-provider runs a JavaScript HTTP server (Node.js or Deno) that solves YouTube's BotGuard attestation challenge and generates GVS PO Tokens on demand. yt-dlp calls this sidecar before each download via the `youtubepot-bgutilhttp` extractor argument. The `web` client with a valid PO token returns standard DASH adaptive formats including audio-only streams (m4a, webm opus). Without the sidecar, `web` client returns SABR-only formats that yt-dlp cannot extract as static files.

This is the mechanism that produced the one confirmed successful download in this project.

### Requirements

- bgutil-ytdlp-pot-provider v1.3.1 (latest, released 2026-03-07)
- yt-dlp >= 2025.05.22 (already satisfied on Railway)
- Node.js >= 20 or Deno >= 2.0.0 inside the container
- Port 4416 accessible within Railway's internal network (no public exposure)
- Cookies still required — `web` client uses them for format unlock

### Pipeline Integration — What Changes in pipeline.py

Nothing. The `bgutil_base_url` parameter already exists in both `check_duration()` and `download_audio()`. The sidecar is wired already:

```python
# Already in pipeline.py — no code changes needed:
extractor_args["youtubepot-bgutilhttp"] = [f"base_url={bgutil_base_url}"]
```

The only missing piece is the deployment of the bgutil service on Railway and setting the environment variable `BGUTIL_BASE_URL` to the Railway-internal URL (e.g., `http://bgutil.railway.internal:4416`).

### Deployment on Railway

Railway supports Docker-based services independently of Compose. The bgutil image is on Docker Hub as `brainicism/bgutil-ytdlp-pot-provider:latest`.

1. Add a new Railway service using the Docker Hub image.
2. Expose port 4416 on Railway's internal network only (no public domain needed).
3. Set `BGUTIL_BASE_URL=http://<service-name>.railway.internal:4416` in the main app's Railway environment variables.
4. No changes to `railway.toml` for the main app service.

### Complexity: MEDIUM

- ~0 lines of code change in pipeline.py (already supported)
- 1 new Railway service (Docker image, no custom Dockerfile)
- 1 new environment variable

### Current Status (Confidence: HIGH)

Project has 528 GitHub stars, 155 commits, 29 releases. Latest release 1.3.1 on 2026-03-07. Actively maintained. Docker Hub image confirmed available. This is the approach explicitly documented in the official yt-dlp PO Token Guide wiki. The project requires yt-dlp >= 2025.05.22; Railway's yt-dlp is already current enough.

### Known Limitation: TypeScript Version Has Memory Leak

The Node.js/TypeScript implementation accumulates +25MB per request due to V8 isolate management issues. For Railway's hobby plan (512MB RAM), this means the service needs periodic restarts under load. The Rust variant (Approach 2) eliminates this.

### Recommendation

**This is the primary fix.** Deploy this first.

## Approach 2: bgutil-rs (Rust Sidecar)

### Mechanism

Functionally identical to Approach 1 — provides PO tokens via HTTP server on port 4416 using the same API contract — but the server is implemented in Rust instead of Node.js/TypeScript. Docker image: `jim60105/bgutil-pot`.

### Performance Comparison

| Metric | Node.js (Approach 1) | Rust (Approach 2) |
|--------|---|---|
| Memory per request | +25MB (leak) | Stable ~80MB baseline |
| Token generation speed | Baseline | ~3x faster |
| Architecture | V8/JSDOM emulation | Worker thread isolation |
| Docker image size | Larger (Node runtime) | Smaller (compiled binary) |

### Complexity: MEDIUM (drop-in replacement for Approach 1)

Same HTTP API on port 4416, same yt-dlp extractor args. Swap Docker image reference only.

### Current Status (Confidence: MEDIUM)

Newer project (fork of the original). Pre-compiled binary for Linux/amd64. Fewer community reports than the Node.js version. No known issues specific to Railway.

### Recommendation

**Use as upgrade to Approach 1 if Railway memory metrics show the Node.js leak.** Start with Approach 1 (more battle-tested), migrate to this if needed.

## Approach 3: iOS Client (`player_client=ios`)

### Mechanism

The iOS client uses a different YouTube API endpoint and historically bypassed some PO token requirements for format fetching.

### Critical Limitation: Cookies Are Silently Ignored

The iOS client silently discards any `cookiefile` passed to yt-dlp. There is no error message — the download proceeds as an unauthenticated public request. Per a documented analysis: "the iOS client simply ignores it. For age-restricted videos, private videos, or accounts that have verified status, this means your cookies are silently dropped and you get a public-access response (or a rejection)."

This is the confirmed "iOS cookie trap" with a dedicated DEV Community article documenting it.

### Datacenter IP Behavior

Unauthenticated requests from datacenter IPs with the iOS client can return format lists, but without authentication the quality and availability is limited. The GVS PO token is still required for some iOS client requests. More importantly, in early 2026, YouTube began forcing SABR streaming on iOS clients in some configurations, reducing available audio-only formats.

### Format Availability in 2026

Multiple open GitHub issues report that DASH audio-only formats (140, 251, etc.) are disappearing even with iOS client as YouTube forces SABR across more clients. This is not a stable configuration.

### Complexity: LOW

One-line change: `player_client=ios` replacing `android`.

### Current Status (Confidence: LOW)

Documented cookie trap confirmed. SABR forcing on iOS reported in open GitHub issues (still unresolved as of 2026-05-10). The previous cookie-free fallback (`tv_embedded`) was removed in January 2026.

### Recommendation

**Not recommended as primary approach.** The SoundGrabber use case requires cookies for consistent format access from datacenter IPs. iOS client drops those cookies silently. At best, this is a last-resort unauthenticated fallback for public videos — which is worse than what the current android path already does.

## Approach 4: TV Clients (tv, tv_embedded, tv_simply)

### tv_embedded — REMOVED

`tv_embedded` was removed from yt-dlp entirely in January 2026 via PR #15726. It was the historical recommendation for bypassing PO token requirements. It does not exist in current yt-dlp versions. Do not use.

### tv — BROKEN since January 16, 2026

YouTube began requiring authentication for the TV client starting January 16, 2026 (GitHub issue #15583). The client returns `playability status: LOGIN_REQUIRED` even when cookies are provided. This is a server-side YouTube decision — cookies passed to yt-dlp do not satisfy the TV client's new auth requirements. Users confirmed: "Starting January 16, 2026, the TV player formats are blocked behind a login requirement."

### tv_simply — Insufficient documentation

Per the official PO Token Guide, `tv_simply` does not require a PO token. However, there is no documentation on whether it works from datacenter IPs, and given the TV client family trend (progressively locked behind login), confidence is LOW.

### Complexity: LOW (but irrelevant — approaches are broken)

### Recommendation

**Do not use any TV client variant.** `tv_embedded` removed. `tv` broken since January 2026. `tv_simply` undocumented for datacenter use and likely subject to the same lockdown trajectory.

## Approach 5: Cloudflare WARP Proxy

### Mechanism

Cloudflare WARP is a free VPN product routing traffic through Cloudflare's global network. YouTube does not flag Cloudflare IP ranges as datacenter bots (these IPs are also used by residential WARP users). The approach requires running three containers: a Wireguard client, the wgcf CLI tool to configure the tunnel, and a Squid proxy to expose the tunnel as an HTTP proxy. yt-dlp is then configured with `--proxy`.

### Confirmed Working?

One confirmed working report (Oracle Cloud datacenter). The author explicitly states: "We don't know if this solution is stable over time" — YouTube could add Cloudflare IP ranges to its blocklist at any point.

### Railway Compatibility

Poor. Railway's PaaS model runs individual services, not Docker Compose stacks. The WARP approach requires 3 tightly networked containers sharing a Wireguard tunnel. On Railway, this would require either:
- A single container running all three processes (systemd-like, anti-pattern for PaaS)
- Three Railway services with complex networking that may not support the Wireguard kernel module requirement

Wireguard typically requires kernel-level network capabilities that PaaS environments restrict.

### Complexity: HIGH

- 3 additional processes or containers
- Custom Dockerfile modifications
- Network routing configuration
- Kernel networking capability requirements uncertain on Railway
- Potentially broken by YouTube at any time

### Recommendation

**Not recommended for Railway.** Too complex, unstable, and architecturally incompatible with Railway's service model. bgutil solves the same problem more cleanly at lower risk.

## Approach 6: Residential Proxy

### Mechanism

Route all yt-dlp HTTP requests through a residential proxy service (Bright Data, Oxylabs, Smartproxy, etc.). Residential IPs are assigned by ISPs to actual households — YouTube does not flag them. yt-dlp supports proxies via the `proxy` option in `ydl_opts`.

### Does It Work?

Yes — this is the only approach confirmed to solve the problem at the **network layer** without any YouTube-side workaround. From YouTube's perspective, the request originates from a home ISP. PO token requirements still apply depending on client choice.

### Cost Analysis for SoundGrabber

- Pay-as-you-go: $3.50/GB (Bright Data)
- Monthly plan: $12 for 2GB
- Typical 10-minute YouTube audio download: ~30–80MB audio + handshaking overhead ≈ $0.11–$0.28 per download
- At 100 downloads/day: ~$11–$28/day, or $330–$840/month
- SoundGrabber is a **free tool** — this cost structure is not viable

### Complexity: MEDIUM

- No code changes in pipeline.py beyond adding `proxy` to `ydl_opts`
- Credential management via Railway environment variable
- Ongoing cost monitoring required
- Proxy service reliability varies by provider

### Recommendation

**Not recommended for SoundGrabber's use case.** The tool is free for users, making recurring bandwidth costs at this scale prohibitive. This is the correct answer for commercial scrapers at scale, not for a community tool. bgutil achieves comparable reliability at $0.

## Approach 7: Cookie Refresh Automation

### What This Is

Not a bypass — a maintenance approach. YouTube cookies expire every 2 weeks (sometimes 3–5 days if the session remains open in a browser). If cookies expire, all approaches requiring cookies will fail silently or return bot-detection errors.

### Auto-Refresh Landscape in 2026

No viable automated cookie refresh exists for server environments:

- **Chrome extraction**: Broken since Chrome 127 (July 2024). App-bound encryption prevents external programs from accessing cookies, even with admin rights.
- **Firefox extraction**: Works (plain SQLite, no encryption) but requires a real Firefox session with a logged-in YouTube account. No headless Firefox path generates valid YouTube session cookies.
- **yt-dlp `--cookies-from-browser`**: Only works if a browser profile exists on the same machine. Railway has no browser installed.
- **OAuth2**: Dead since 2024.

### Cookie Lifespan

- Fresh private/incognito session: ~2 weeks before expiry
- Active session (browser still open): 3–5 days due to rotation
- Best practice: export from a private window that is never reopened — this maximizes lifespan to ~2 weeks

### Operational Impact

Manual cookie refresh every 2 weeks is required regardless of which bypass approach is chosen. This is an operational maintenance item, not a code fix. The `YTDLP_COOKIES_B64` Railway environment variable needs to be updated with a fresh base64-encoded cookies.txt every ~2 weeks.

### Complexity: HIGH (for full automation), LOW (for monitoring + alerting)

Full automation requires headless browser + Playwright + YouTube login in the Railway environment — significant overhead and Google actively fights this.

A lightweight alternative: add a health check that runs a yt-dlp metadata-only fetch on a known stable video every hour. If it fails, send an alert. This detects cookie expiry without attempting automation.

### Recommendation

**Implement cookie expiry monitoring, not auto-refresh.** A metadata-only probe (`skip_download=True`) costs negligible bandwidth and catches both cookie expiry and bgutil failure. Manual refresh cycle (2 weeks) is acceptable for v1.2.

## Approach 8: youtube-oauth2 Plugin

### Status: DEAD

The `yt-dlp-youtube-oauth2` plugin is explicitly marked **OBSOLETE** on its GitHub repository. Native OAuth2 support was added to yt-dlp in 2024.10.22 using YouTube TV's device code flow, but Google disabled this authentication endpoint shortly after. The official yt-dlp FAQ now states: "OAuth login no longer works with YouTube extractors. Use `--cookies` instead."

### Recommendation

**Do not use.** Plugin is archived. The underlying mechanism (YouTube TV device code auth) is server-side disabled by Google.

```
Valid cookies in YTDLP_COOKIES_B64 (manual 2-week refresh)
  → bgutil sidecar (Railway service, port 4416 internal)
    → web client + PO token
      → DASH audio-only format selection
        → FFmpegExtractAudio postprocessor (WAV)
          → analyze_audio() — BPM + key
```

The chain breaks at step 2 (bgutil sidecar deleted from Railway). Everything above and below it is functional.

## MVP Fix: What v1.2 Requires

The code is complete. `pipeline.py` already handles `bgutil_base_url` in both `check_duration()` and `download_audio()`. The fix is:

1. **Deploy bgutil as a Railway service** — use Docker image `brainicism/bgutil-ytdlp-pot-provider:latest`, expose port 4416 on internal network only.
2. **Set `BGUTIL_BASE_URL` environment variable** in the main app — value: the Railway-internal service URL on port 4416.
3. **Verify cookie freshness** — confirm `YTDLP_COOKIES_B64` contains cookies exported within the last 2 weeks from a private browser session.
4. **Add cookie expiry health check** — a periodic metadata-only yt-dlp probe that alerts when the pipeline is broken before users hit it.

Estimated scope: 1 Railway service + 1 env var + 1 health check route. Zero changes to `pipeline.py`.
